# Exercise 09: Bradley-Terry reward model on synthetic preferences

Goes with the lecture on [reward modeling](../../notes/lectures/09-reward-modeling.md).

You'll train a small MLP reward model on synthetic preference data using the Bradley-Terry NLL. No LLM, no transformer, no GPU: just 4-dim feature vectors and a 3-layer net. The point is to see the BT objective working end-to-end and to convince yourself that pairwise preference labels really do identify the underlying reward ranking.

## The setup

There's a hidden reward function `r*(x) = w*·x` for a fixed 4-dim weight vector `w*`. Each "response" is a 4-dim feature vector (think: quality, length, formality, accuracy; any four latent dims). The reward model never sees `w*`; all it sees are labeled pairs.

A preference pair `(chosen, rejected)` is generated by:

1. sampling two random feature vectors `a` and `b`,
2. computing the true rewards `r*(a)` and `r*(b)`,
3. labeling `a` as chosen with probability `σ(β · (r*(a) - r*(b)))`.

That last step *is* the Bradley-Terry model; we're labeling data with the same distribution the model assumes during training. Higher `β` means cleaner labels; this exercise uses `β = 4.0`, which is firm but not deterministic.

The model is trained on the NLL of the labels under its own predicted rewards:

```
L = -log σ(r_chosen - r_rejected)
```

averaged over a minibatch.

Success means the trained model's predicted reward ordering on a held-out set of feature vectors matches the true reward ordering, measured by Spearman rank correlation. A random model scores around 0; a perfect-ranking model scores 1.0. The acceptance threshold is 0.85.

## Why this matters

In real RLHF, the reward model isn't given the true reward, only pairwise human labels. The reason the pipeline works at all is that pairwise labels are enough to identify a reward function up to a monotone transform, which is enough for ranking, which is all the RL stage needs. This exercise is the toy version of that claim: we set up a known true reward, train only on pairwise labels, and check that the trained model can rank held-out inputs correctly. Same loss, same generalization mechanism, just no language model in the way.

## The task

Fill in the TODOs in [`starter.py`](./starter.py). Three pieces:

1. `RewardModel`: a small MLP. Single input `(FEATURE_DIM,)` returns a scalar `()`; batched input `(B, FEATURE_DIM)` returns `(B, 1)`.
2. `bradley_terry_loss(chosen_rewards, rejected_rewards)`: `-log σ(chosen - rejected)`, mean over the batch. Use `F.logsigmoid` for numerical stability.
3. `train_reward_model(...)`: the loop: sample a minibatch, forward, BT loss, backward, step. Track per-step training loss and per-step held-out Spearman correlation.

The data generator, the held-out evaluator, and the Spearman helper are given to you so you can focus on the model and the loss.

- `test_reward_model_forward`: single input gives a scalar tensor; batched input gives `(B, 1)`.
- `test_bradley_terry_loss_perfect_separation`: chosen rewards much higher than rejected → loss ≈ 0.
- `test_bradley_terry_loss_tied`: chosen == rejected → loss = `log(2) ≈ 0.6931`.

You're done when the tests pass and you can explain:

- Why is the loss for tied rewards exactly `log(2)`?
- The true reward is linear in the features, but the model is a 2-hidden-layer MLP. Why does it still recover the ranking? (Hint: think about what monotone transforms preserve.)

## Going further (optional)

- Replace the linear `r*(x) = w*·x` with a nonlinear true reward (say `tanh` of a linear combination, or a small fixed random MLP). Does the model still recover the ranking? Does the final Spearman drop?
- Lower `β` to 1.0, then 0.3. Plot the held-out Spearman at the end as a function of `β`. This is the label-noise curve; RLHF datasets in the wild are not at `β = 4.0`.
- Track the per-step `losses` and `spearman` lists. Loss can keep falling after Spearman has plateaued, because the model is sharpening its scores rather than re-ordering. That's the gap between optimizing the loss and improving the actual ranking, relevant when you read about reward overoptimization in lecture 09.
- Add a length bias on purpose: in `make_preference_dataset`, label longer feature vectors (`|x|`) as chosen with extra probability. The model will learn this spurious feature. This is the toy version of length-bias reward hacking, also covered in lecture 09.
